# HILP (Human-in-the-Loop Protocol) チェックポイント

## エスカレーション判断フロー

```
意思決定が必要
  ├── Level 1 (Auto-Approve)? → 即実行
  ├── Level 2 (Propose & Execute)? → 関連役職に共有後実行
  ├── Level 3 (HILP Required)? → CPOにエスカレーション
  └── Level 4 (CPO Only)? → CPOに委譲
```

---

## エスカレーション必須トリガー

### 財務
- [ ] 月額1万円超の新規支出
- [ ] 既存サービスのプラン変更（アップグレード/ダウングレード）
- [ ] 価格設定の新規作成・変更
- [ ] 返金・クレジット対応（個別判断が必要な場合）

### 技術
- [ ] 不可逆なインフラ変更（DB移行、ドメイン変更）
- [ ] 新規外部サービス・APIの導入
- [ ] ユーザーデータスキーマの破壊的変更
- [ ] セキュリティインシデント発生時

### 戦略
- [ ] ターゲット顧客セグメントの変更
- [ ] ビジネスモデルの変更
- [ ] 新規プロダクトライン追加
- [ ] 既存機能の廃止
- [ ] 新規事業（BU）の立ち上げ提案
- [ ] 既存事業の撤退・ピボット判断
- [ ] 事業間リソース配分の変更

### コミュニケーション
- [ ] ユーザー向け重要告知（障害報告、料金変更通知）
- [ ] 外部パートナーとの交渉開始
- [ ] プレスリリース・公式声明

### エクイティ（ファントムストック）
- [ ] 個別付与（11ユニット以上）
- [ ] キャッシュアウト承認
- [ ] プール上限変更
- [ ] バリュエーション方式変更
- [ ] ベスティング条件変更
- [ ] 没収決定
- [ ] 契約ひな型の承認・変更（Level 4）
- [ ] 法人化移行方針（Level 4）

### 情報分類
- [ ] 格上げ（Internal → Confidential）
- [ ] 格下げ（全パターン — 不可逆な公開判断）
- [ ] ポリシー重大改訂
- [ ] 新分類レベル追加

## エスカレーション提示フォーマット

```markdown
## 🔔 HILP エスカレーション

**発信元**: [役職名]
**カテゴリ**: 財務 / 技術 / 戦略 / コミュニケーション

### コンテキスト
[現在の状況と、なぜこの判断が必要になったか]

### 選択肢
| # | 選択肢 | メリット | デメリット | コスト影響 |
|---|--------|---------|-----------|-----------|
| 1 | [A案] | ... | ... | ¥X/月 |
| 2 | [B案] | ... | ... | ¥Y/月 |
| 3 | [C案] | ... | ... | ¥Z/月 |

### 推奨
**[推奨案]** を推奨。理由: [データに基づく根拠]

### 緊急度
🔴 **高**: 24時間以内に判断が必要
🟡 **中**: 1週間以内に判断が必要
🟢 **低**: 次回ボード会議で議論可能

### 影響範囲
- **影響部門**: [CEO/CTO/CFO/CMO/COO]
- **影響KPI**: [具体的なKPI名]
- **影響ユーザー数**: [推定]

### 期限
[この判断を行うべき期日とその理由]
```

## 応答フォーマット

CPOの判断後、以下を記録する:

```markdown
### HILP Resolution
- **判断日**: YYYY-MM-DD
- **CPO決定**: [選択した案]
- **補足指示**: [あれば]
- **次のアクション**: [担当役職: タスク]
```

---

## 緊急度ガイドライン

| 緊急度 | 定義 | 例 |
|--------|------|-----|
| 🔴 高 | サービス停止、セキュリティ侵害、法的リスク | インシデント対応、データ漏洩 |
| 🟡 中 | 機会損失の可能性、計画への影響 | 競合動向への対応、価格改定 |
| 🟢 低 | 改善提案、中長期戦略 | 新機能提案、ツール変更 |
